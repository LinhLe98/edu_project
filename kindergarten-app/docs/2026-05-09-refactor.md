# 2026-05-09 — Refactor & Feature Fixes

## Summary
Structural improvements and bug fixes across the Angular kindergarten app. All changes pass `ng build` with zero errors.

---

## Feature Fixes

### 1. Slideshow `ctaLink` fragment navigation
**Problem:** `slide-4` had `ctaLink: '/gioi-thieu#teacher-groups'` passed directly to `[routerLink]`. Angular Router treats the hash as part of the path, not a fragment, so the link navigated nowhere meaningful.

**Fix:**
- Added optional `ctaFragment?: string` field to `HeroSlide` model (`slide.model.ts`)
- Updated `slides.data.ts`: `slide-4` now uses `ctaLink: '/gioi-thieu'` + `ctaFragment: 'teacher-groups'`
- Updated `slideshow.component.html` to bind `[fragment]="slide.ctaFragment"` on the CTA anchor

### 2. About page — teacher group display limit
**Problem:** `buildGroupMembers()` in `teacher-groups.component.ts` returned the group leader plus *all* vice-leaders with no upper bound, violating the 1 leader + 2 vice-leaders display requirement.

**Fix:** Added `.slice(0, 2)` on the vice-leaders array before building the member list.

### 3. Detail page related-item navigation
**Problem:** `NewsDetailComponent`, `TeacherDetailComponent`, and `ActivityDetailComponent` used `this.route.snapshot.paramMap.get()` in `ngOnInit`. Angular reuses the same component instance when navigating between items on the same route pattern (e.g., `/tin-tuc/:slug`), so `ngOnInit` does not re-fire. Clicking a related article/colleague/program silently left the page stale.

**Fix:** Replaced snapshot reads with `this.route.paramMap` observable subscriptions using `takeUntilDestroyed(this.destroyRef)`. Each navigation to a different slug/id now triggers a fresh data load and updates all signals.

Files changed:
- `src/app/pages/news/news-detail.component.ts`
- `src/app/pages/teacher/teacher-detail.component.ts`
- `src/app/pages/activities/activity-detail/activity-detail.component.ts`

### 4. Learning materials download
**Problem:** `downloadMaterial()` created an `<a download>` element for a cross-origin PDF URL. Browsers ignore the `download` attribute for cross-origin resources, so no download dialog appeared.

**Fix:** Replaced the direct `<a>` approach with `fetch()` → `Blob` → `createObjectURL()` pipeline. If the fetch fails (e.g., CORS), the method falls back to `window.open(url, '_blank')` so the user can still access the file.

File changed: `src/app/pages/library/learning-materials/learning-materials.component.ts`
